# “寻迹校园”自然语言交互改进设计

日期：2026-07-07  
范围：离线 MockLLM 自然语言解析、Agent 工具选择、前端自由输入体验和回归测试

## 1. 目标

当前 Web 页面和 README 已提供 `/api/chat`，技术上可以提交任意 `message`，但 demo 的稳定路径主要依赖固定快捷按钮和 MockLLM 中分散的正则规则。用户一旦换成更口语化的表达，例如“我昨天在图书馆二楼丢了个黑色蓝牙耳机”或“CL0001 现在处理到哪了”，离线教学桩可能无法正确路由、抽取实体或选择工具。

本次改进目标是：在不配置 API Key、无网络依赖的情况下，让用户可以直接使用自定义中文自然语言完成寻物、认领进度查询、交接时段查询和规则咨询。快捷按钮保留为示例入口，但自由输入成为主要交互方式。

## 2. 非目标

- 不接入新的外部 NLP、向量库或前端框架。
- 不改变三个微服务的 HTTP 契约。
- 不重写 BPMN 认领流程，也不绕过现有护栏。
- 不把离线 MockLLM 扩展成通用大模型；只覆盖校园失物招领 demo 的核心表达。
- 不新增账号登录、数据库或持久化状态。

## 3. 推荐方案

采用“轻量自然语言解析器 + MockLLM 复用”的方案。

在 `llm.py` 内新增一个小型解析函数，例如 `_parse_user_text(text)`，返回统一结构：

```python
{
    "intent": "寻物",
    "item_id": "LF2026001",
    "claim_id": "CL0001",
    "location": "图书馆",
    "color": "黑色",
    "category": "耳机",
    "wants_handover": False,
    "wants_status": False,
    "wants_policy": False,
}
```

MockLLM 的 `_route()`、`_intent()`、`_tool_step()` 和必要的最终回答逻辑都读取该统一结果，避免每个分支各自维护一套关键词。这样改动集中，测试也能直接证明自然语言理解能力。

## 4. 文件边界

| 文件 | 职责 |
|---|---|
| `llm.py` | 新增统一解析函数，增强 MockLLM 路由、意图 JSON、工具选择和口语化表达覆盖 |
| `tests/test_tools_agent.py` | 覆盖自由寻物、认领进度、交接时段和规则咨询的 Agent/MockLLM 行为 |
| `tests/test_experiment4.py` | 覆盖 Web 页面自由输入提示和快捷按钮仍可用 |
| `web/index.html` | 调整欢迎语、输入框占位符和快捷区文案，使自由输入是第一路径 |
| `README.md` | 说明 demo 支持自由自然语言输入，并列出离线可复现样例 |

`agent.py`、`app.py`、`web_server.py`、三个微服务和 BPMN 处理器原则上不改。若测试显示 `agent.py` 只需增加很小的上下文提示，可以纳入本次范围，但不得改变对外接口。

## 5. 自然语言覆盖

### 5.1 寻物

应支持用户不使用固定按钮时的口语化描述：

- “我昨天在图书馆二楼丢了个黑色蓝牙耳机”
- “图书馆有没有捡到黑色耳机”
- “帮我找一下食堂的校园卡”

解析规则：

- 地点识别仍归一到现有演示地点：`图书馆`、`教学楼`、`食堂`、`操场`、`宿舍`。
- “二楼”“服务台”“昨天”等细节可以保留在原始请求中供回复使用，但搜索工具只传已有契约支持的 `keyword` 和 `location`。
- “蓝牙耳机”“耳麦”等归一为 `耳机`；“电脑”“笔记本”归一为 `笔记本电脑`。

### 5.2 认领与进度查询

认领流程继续由现有 BPMN 处理。包含 `LF...` 且表达“认领、证明、失主、我的”等诉求时，仍进入 `expert_claim()` 的 BPMN 路径。

认领单查询应支持：

- “想查 CL0001 现在处理到哪了”
- “CL0001 审核结果出来了吗”
- “我的认领单 CL0001 是什么状态”

这些请求应路由到认领专家，并由 ReAct 调用 `query_claim({"claim_id": "CL0001", "user_id": 当前用户})`。授权护栏继续在 `app.py` 中先行拦截越权访问。

### 5.3 交接

应支持：

- “LF2026001 可以什么时候去取”
- “这个耳机在哪里领取”
- “帮我看下 LF2026001 的交接时间”

当文本包含 `LF...` 且表达“取、领取、交接、预约、时段、什么时候、哪里”等诉求时，路由到交接专家，并调用 `query_item` 后调用 `list_handover_slots`。如果没有 `LF...`，则返回当前已有的能力说明或候选搜索结果，不新增模糊预约流程。

### 5.4 规则咨询

应支持：

- “电脑这种贵重物品为啥要人工审核”
- “高价值物品为什么不能直接领”
- “隐私信息会不会被泄露”

这些请求路由到规则专家或在 ReAct 中调用 `search_policy`，输出仍来自现有 RAG 政策文本。

## 6. 前端交互

前端继续是单页 HTML，不引入构建工具。

调整重点：

- 欢迎语改为提示用户可直接描述遗失物、编号、认领单或规则问题。
- 输入框占位符改为自由输入样例，例如“例如：我昨天在图书馆二楼丢了黑色蓝牙耳机”。
- 快捷按钮区域文案改为“示例”，保留课程演示按钮，但视觉和文案上不再暗示只能点击按钮。
- 继续使用 `textContent` 渲染用户输入、回复和 trace，避免 XSS 风险。

## 7. 错误处理与安全

- 输入护栏仍在 `app.py` 编排前执行，提示注入、隐藏特征索取、流程变量篡改和审核绕过不进入 Agent。
- 认领单和预约编号仍通过授权护栏校验当前 `user_id`。
- 自然语言解析只生成公开工具参数，不生成隐藏特征、`match_score` 或 `high_value`。
- 对无法识别的自由输入，保持现有“我可以帮您查找失物、提交认领申请、查询交接或解释认领规则”的受控回复。

## 8. 测试策略

采用测试先行：

1. 先在 `tests/test_tools_agent.py` 增加失败测试，证明自由表达目前覆盖不足。
2. 实现 `_parse_user_text()` 并让 MockLLM 分支复用它。
3. 增加 Web 静态页面测试，确认输入框和欢迎语鼓励自由输入，同时快捷示例仍存在。
4. 运行完整回归：

```powershell
python -B -m unittest discover -s tests -v
python -B -m compileall -q .
git diff --check
```

## 9. 验收标准

- 无 API Key 时，以下输入可以稳定返回符合预期的离线结果：
  - “我昨天在图书馆二楼丢了个黑色蓝牙耳机”返回候选 `LF2026001`。
  - “想查 CL0001 现在处理到哪了”在授权用户下查询认领单状态。
  - “LF2026001 可以什么时候去取”返回失物信息和交接时段。
  - “电脑这种贵重物品为啥要人工审核”返回高价值物品人工复核规则。
- 快捷按钮仍可演示原有寻物、三条认领路径、RAG、注入测试和越权测试。
- 护栏、BPMN、RAG、微服务和 Web API 既有测试继续通过。
